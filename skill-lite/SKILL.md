# How to Say Lite — 快速表达润色

## Overview

不需要设定，不需要画像，直接润色你的表达。
贴原文 → 诊断问题 → 给出改写。30 秒搞定。

## 使用方式

用户贴一段文字或描述一个场景，按以下流程处理：

### 1. 快速识别

判断用户需求属于哪种：
- **直接润色**：用户贴了一段话说"帮我改改" → 直接进入诊断
- **场景准备**：用户说"明天要汇报" → 问清场景后给结构模板
- **一句话求助**：用户问"这句话怎么说" → 直接给改写版本

### 2. 诊断 & 改写

对用户的表达，逐条检查以下 10 条铁律：

1. **先说结论。** — 第一句是不是结论？不是就调到前面。
2. **一次只讲一个主结论。** — 有没有塞了多个观点？拆开。
3. **最多 2-3 个支撑点。** — 支撑太多就砍。
4. **先想听众关心什么。** — 内容是对着听众的需求，还是自说自话？
5. **例子服务观点。** — 例子是支撑结论的，还是喧宾夺主？
6. **少说感受，多说判断。** — "我觉得"改成"我的判断是"。
7. **少说抽象词，多说具体。** — "赋能"改成"把审批从5步减到2步"。
8. **说不清就停，不要硬撑。** — 有没有越说越乱的部分？
9. **汇报讲价值，不只讲过程。** — 有没有只说做了什么，没说值不值？
10. **让人记住一个点。** — 全篇最关键的一句话是什么？

### 3. 输出格式

```
【原文】用户的原始表达
【问题】违反了哪条铁律、具体哪里有问题
【改写】按铁律重构后的版本
【铁律】引用的铁律编号
```

如果原文有多个问题，逐条诊断，每条单独给改写。

### 4. 场景速查

用户描述场景时，快速给结构模板：

| 场景 | 结构模板 |
|------|---------|
| 向上汇报 | 进度 → 问题 → 判断 → 需要的支持 |
| 开会发言 | "我补充一个X" → 一句结论 → 一个支撑 |
| 面试回答 | 结论 → 过程 → 结果 → 反思 |
| 自我介绍 | 我是谁 → 我擅长什么 → 我能贡献什么 |
| 提问 | 我的理解 → 我的疑问 → 想确认的问题 |
| 回答问题 | 先给结论 → 分层展开 → 收束落地 |
| 1-on-1 | Wins → Blockers → Asks |
| Quarterly Review | Impact statement → Data → Growth areas |

## 12 句救急句型

紧张时直接套用：

1. 我的结论是……
2. 这个问题我讲两点。
3. 先说结果，再补充原因。
4. 核心问题不在 A，而在 B。
5. 我补充一个风险。
6. 我有一个不同判断。
7. 从执行角度看，我更倾向于……
8. 这个方案可行，但前提是……
9. 如果今天要推进，我建议先做……
10. 我收一下，我真正想表达的是……
11. 这件事目前最大的卡点是……
12. 我现在没有完整答案，但初步判断是……

## 8 条红线

改写时必须消除这些问题：

| 问题 | 替代 |
|------|------|
| 边想边说太久 | 停顿 2 秒，用救急句型开口 |
| 铺垫超过 30 秒 | 第一句就给结论 |
| 一段塞多个观点 | 一段只干一件事 |
| "我觉得"开头 | "我的判断是" |
| 句尾飘着结束 | "核心就是这两点" |
| 过度自我否定 | "我先给一个当前理解" |
| 堆砌形容词 | 给事实和比较数据 |
| 为礼貌模糊重点 | 可以温和，但必须清楚 |
